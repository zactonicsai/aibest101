Prerequisites

Before you begin

You don't need to know Elasticsearch — that's what this tutorial is for. But you do need a few tools installed locally. Every example in this guide assumes the following are on your PATH:

  • Docker & Docker Compose — for running Elasticsearch in a container (Compose v2 plugin is fine)
  • Java 21 — any LTS distribution (Temurin, Zulu, Corretto). Verify with java --version
  • Go 1.22+ — verify with go version
  • Python 3.11+ — verify with python3 --version
  • curl — we'll poke the REST API directly from the shell
About versions. This tutorial targets Elasticsearch 9.3.3 (the latest stable release as of April 2026). The official clients for each language are versioned to match: elasticsearch-java 9.x, go-elasticsearch/v9, and elasticsearch-py 9.x. Always match your client's major version to your cluster's.

Step Two

Verify the cluster is alive

Before writing a single line of application code, confirm the cluster responds. Two endpoints matter: the root (banner + version) and _cluster/health.

shell 
curl -s http://localhost:9200 | jq .
curl -s http://localhost:9200/_cluster/health | jq .

You should see a JSON blob that includes a version.number of 9.3.3 and a cluster status of green (single-node clusters with no replicas are green by default).

Three cluster statuses you'll see: green — all primary & replica shards active. yellow — primaries active, some replicas missing (normal on dev). red — at least one primary shard is unavailable. Something is wrong.
03
Step Three

Indexes, documents, shards, mappings

Don't skip this. If these four words don't click, the rest of Elasticsearch feels like magic — and magic is what causes bad production incidents at 3 a.m.

Index

A logical collection of similar documents. Roughly analogous to a table in a relational database — but much more flexible about what "a row" can look like.

Document

A JSON object stored in an index. Every document has a unique _id (auto-generated if you don't supply one) and is internally indexed into an inverted index.

Shard

A physical unit of storage. Each index is split across one or more primary shards, each of which can have replicas. Shards are Lucene indexes under the hood.

Mapping

The schema. It tells Elasticsearch how each field in your document should be indexed: as full-text, a keyword, a number, a date, a geo-point, a vector, etc.

The mapping is the single most consequential decision you make. Elasticsearch will happily infer one for you the first time you index a document (this is called dynamic mapping), but the result is often wrong — e.g., strings defaulting to both text and keyword, which doubles your storage for no reason. Pros always declare mappings explicitly.

Step Four

Design and create the index

Our example dataset will be a catalog of books. Each book has a title (searchable full-text), an author name (often searched as an exact value), a publish date, a price, a list of tags, and a short description.

We'll create the index books with an explicit mapping. Run this against the cluster:

shell — create index with mapping 
curl -X PUT http://localhost:9200/books \
  -H "Content-Type: application/json" \
  -d '{
    "settings": {
      "number_of_shards": 1,
      "number_of_replicas": 0,
      "refresh_interval": "1s",
      "analysis": {
        "analyzer": {
          "english_folded": {
            "type": "custom",
            "tokenizer": "standard",
            "filter": ["lowercase", "asciifolding", "english_stop", "english_stemmer"]
          }
        },
        "filter": {
          "english_stop":    { "type": "stop",     "stopwords": "_english_" },
          "english_stemmer": { "type": "stemmer",  "language": "english"  }
        }
      }
    },
    "mappings": {
      "dynamic": "strict",
      "properties": {
        "title":       { "type": "text", "analyzer": "english_folded",
                         "fields": { "raw": { "type": "keyword" } } },
        "author":      { "type": "keyword" },
        "description": { "type": "text", "analyzer": "english_folded" },
        "tags":        { "type": "keyword" },
        "price":       { "type": "scaled_float", "scaling_factor": 100 },
        "published":   { "type": "date", "format": "strict_date_optional_time" },
        "in_stock":    { "type": "boolean" },
        "ingested_at": { "type": "date" }
      }
    }
  }'

A lot is happening here. The table below explains every choice:

Field Type Why
titletext + keywordFull-text search on the stemmed form, exact match / sorting via title.raw
authorkeywordExact-match filter, aggregate by author — no tokenizing needed
descriptiontextSearchable prose; no keyword sub-field needed
tagskeywordArray of exact values for filtering & facets
pricescaled_floatCheaper than double when you only need 2 decimal places
publisheddateISO-8601 strings; supports range queries & date math
ingested_atdateMetadata field — we'll set this on the client side
Why "dynamic": "strict"? Because it rejects any document with a field you didn't declare. That single setting prevents an entire category of production pain: a misspelled field name silently creating a new mapping, bloating your index, and poisoning aggregations. Turn it on early.

package com.example;

import co.elastic.clients.elasticsearch.ElasticsearchClient;
import co.elastic.clients.elasticsearch.core.IndexResponse;
import co.elastic.clients.json.jackson.JacksonJsonpMapper;
import co.elastic.clients.transport.rest_client.RestClientTransport;
import org.apache.http.HttpHost;
import org.elasticsearch.client.RestClient;

import java.time.Instant;
import java.time.LocalDate;
import java.util.List;

public class IndexOneBook {

  // Using a Java 21 record for conciseness and immutability.
  public record Book(
      String title,
      String author,
      String description,
      List<String> tags,
      double price,
      LocalDate published,
      boolean in_stock,
      Instant ingested_at
  ) {}

  public static void main(String[] args) throws Exception {
    var restClient = RestClient
        .builder(new HttpHost("localhost", 9200, "http"))
        .build();

    var transport = new RestClientTransport(restClient, new JacksonJsonpMapper());
    var es = new ElasticsearchClient(transport);

    var book = new Book(
        "The Pragmatic Programmer",
        "Andy Hunt",
        "Classic guide to software craftsmanship.",
        List.of("software", "engineering", "career"),
        39.95,
        LocalDate.of(1999, 10, 20),
        true,
        Instant.now()
    );

    IndexResponse resp = es.index(i -> i
        .index("books")
        .id("pragprog-1")      // stable business id — idempotent ingest
        .document(book)
    );

    System.out.println("Indexed: " + resp.id() + " result=" + resp.result()
        + " version=" + resp.version());

    transport.close();
    restClient.close();
  }
}

cmd/index-one/main.go 
package main

import (
	"bytes"
	"context"
	"encoding/json"
	"log"
	"strings"
	"time"

	"github.com/elastic/go-elasticsearch/v9"
	"github.com/elastic/go-elasticsearch/v9/esapi"
)

type Book struct {
	Title       string    `json:"title"`
	Author      string    `json:"author"`
	Description string    `json:"description"`
	Tags        []string  `json:"tags"`
	Price       float64   `json:"price"`
	Published   string    `json:"published"` // YYYY-MM-DD
	InStock     bool      `json:"in_stock"`
	IngestedAt  time.Time `json:"ingested_at"`
}

func main() {
	cfg := elasticsearch.Config{
		Addresses: []string{"http://localhost:9200"},
		// Username / Password / CACert go here once you enable security.
	}
	es, err := elasticsearch.NewClient(cfg)
	if err != nil {
		log.Fatalf("client: %s", err)
	}

	book := Book{
		Title:       "The Pragmatic Programmer",
		Author:      "Andy Hunt",
		Description: "Classic guide to software craftsmanship.",
		Tags:        []string{"software", "engineering", "career"},
		Price:       39.95,
		Published:   "1999-10-20",
		InStock:     true,
		IngestedAt:  time.Now().UTC(),
	}

	body, _ := json.Marshal(book)
	req := esapi.IndexRequest{
		Index:      "books",
		DocumentID: "pragprog-1",
		Body:       bytes.NewReader(body),
		Refresh:    "false",
	}

	res, err := req.Do(context.Background(), es)
	if err != nil {
		log.Fatalf("index: %s", err)
	}
	defer res.Body.Close()

	if res.IsError() {
		log.Fatalf("error: %s", res.String())
	}

	var r map[string]any
	_ = json.NewDecoder(res.Body).Decode(&r)
	log.Printf("indexed: id=%s result=%s version=%v",
		r["_id"], r["result"], r["_version"])
	// strings import kept for completeness in larger programs:
	_ = strings.TrimSpace
}

Supply your own _id whenever you can. Using a stable business key (ISBN, UUID, primary-key from your source system) makes ingest idempotent: re-running your pipeline updates the same document instead of creating a new one. Step Six

Bulk ingestion — the fast path

Indexing documents one-at-a-time is convenient but it will never keep up with real data volumes. The _bulk API accepts many operations in a single request. On a laptop the difference between single-doc and bulk ingest is commonly 50–100× throughput.

All three official clients ship a helper so you don't have to hand-roll the NDJSON format yourself. The rule of thumb: batches of 1,000–5,000 documents or 5–15 MB per request, whichever comes first. Bigger isn't better — giant bulk requests just create GC pressure and time-outs.

BulkIndexBooks.java — using the BulkIngester helper 
package com.example;

import co.elastic.clients.elasticsearch.ElasticsearchAsyncClient;
import co.elastic.clients.elasticsearch.ElasticsearchClient;
import co.elastic.clients.elasticsearch._helpers.bulk.BulkIngester;
import co.elastic.clients.elasticsearch._helpers.bulk.BulkListener;
import co.elastic.clients.elasticsearch.core.bulk.BulkOperation;
import co.elastic.clients.json.jackson.JacksonJsonpMapper;
import co.elastic.clients.transport.rest_client.RestClientTransport;
import org.apache.http.HttpHost;
import org.elasticsearch.client.RestClient;

import java.time.Duration;
import java.time.Instant;
import java.time.LocalDate;
import java.util.List;

public class BulkIndexBooks {

  public record Book(String title, String author, String description,
                     List<String> tags, double price, LocalDate published,
                     boolean in_stock, Instant ingested_at) {}

  public static void main(String[] args) throws Exception {
    var restClient = RestClient.builder(new HttpHost("localhost", 9200, "http")).build();
    var transport  = new RestClientTransport(restClient, new JacksonJsonpMapper());
    var es         = new ElasticsearchAsyncClient(transport);

    // BulkIngester batches automatically by size, count, or time window.
    try (BulkIngester<String> ingester = BulkIngester.of(b -> b
        .client(es)
        .maxOperations(1_000)             // flush every 1k ops
        .maxSize(5L * 1024 * 1024)        // or every 5 MB
        .flushInterval(2, java.util.concurrent.TimeUnit.SECONDS)
        .listener(new BulkListener<String>() {
          public void beforeBulk(long id, co.elastic.clients.elasticsearch.core.BulkRequest r,
                                 List<String> ctx) {}
          public void afterBulk(long id, co.elastic.clients.elasticsearch.core.BulkRequest r,
                                List<String> ctx,
                                co.elastic.clients.elasticsearch.core.BulkResponse resp) {
            if (resp.errors()) System.err.println("bulk had failures: " + resp.items().size());
          }
          public void afterBulk(long id, co.elastic.clients.elasticsearch.core.BulkRequest r,
                                List<String> ctx, Throwable t) { t.printStackTrace(); }
        })
    )) {
      for (int i = 0; i < 10_000; i++) {
        String id = "book-" + i;
        Book doc  = new Book(
            "Generated Title " + i,
            "Author " + (i % 50),
            "Description for book " + i,
            List.of("generated", "batch-" + (i / 1000)),
            9.99 + (i % 40),
            LocalDate.now().minusDays(i),
            i % 2 == 0,
            Instant.now()
        );
        ingester.add(op -> op.index(idx -> idx.index("books").id(id).document(doc)), id);
      }
    } // auto-close flushes remaining ops

    System.out.println("done.");
    transport.close();
    restClient.close();
  }
}
cmd/index-bulk/main.go — using esutil.BulkIndexer 
package main

import (
	"bytes"
	"context"
	"encoding/json"
	"fmt"
	"log"
	"strings"
	"sync/atomic"
	"time"

	"github.com/elastic/go-elasticsearch/v9"
	"github.com/elastic/go-elasticsearch/v9/esutil"
)

type Book struct {
	Title       string    `json:"title"`
	Author      string    `json:"author"`
	Description string    `json:"description"`
	Tags        []string  `json:"tags"`
	Price       float64   `json:"price"`
	Published   string    `json:"published"`
	InStock     bool      `json:"in_stock"`
	IngestedAt  time.Time `json:"ingested_at"`
}

func main() {
	es, err := elasticsearch.NewClient(elasticsearch.Config{
		Addresses: []string{"http://localhost:9200"},
	})
	if err != nil {
		log.Fatalf("client: %s", err)
	}

	bi, err := esutil.NewBulkIndexer(esutil.BulkIndexerConfig{
		Client:        es,
		Index:         "books",
		NumWorkers:    4,                 // tune to your CPU / cluster
		FlushBytes:    5_000_000,         // 5 MB
		FlushInterval: 2 * time.Second,
	})
	if err != nil {
		log.Fatalf("bulk indexer: %s", err)
	}

	var indexed, failed uint64
	start := time.Now()

	for i := 0; i < 10_000; i++ {
		book := Book{
			Title:       fmt.Sprintf("Generated Title %d", i),
			Author:      fmt.Sprintf("Author %d", i%50),
			Description: fmt.Sprintf("Description for book %d", i),
			Tags:        []string{"generated", fmt.Sprintf("batch-%d", i/1000)},
			Price:       9.99 + float64(i%40),
			Published:   time.Now().AddDate(0, 0, -i).Format("2006-01-02"),
			InStock:     i%2 == 0,
			IngestedAt:  time.Now().UTC(),
		}

		body, _ := json.Marshal(book)
		err := bi.Add(context.Background(), esutil.BulkIndexerItem{
			Action:     "index",
			DocumentID: fmt.Sprintf("book-%d", i),
			Body:       bytes.NewReader(body),
			OnSuccess: func(_ context.Context, _ esutil.BulkIndexerItem, _ esutil.BulkIndexerResponseItem) {
				atomic.AddUint64(&indexed, 1)
			},
			OnFailure: func(_ context.Context, _ esutil.BulkIndexerItem, res esutil.BulkIndexerResponseItem, err error) {
				atomic.AddUint64(&failed, 1)
				if err != nil {
					log.Printf("err: %s", err)
				} else {
					log.Printf("fail: %s: %s", res.Error.Type, res.Error.Reason)
				}
			},
		})
		if err != nil {
			log.Printf("add: %s", err)
		}
	}

	if err := bi.Close(context.Background()); err != nil {
		log.Fatalf("close: %s", err)
	}

	log.Printf("indexed=%d failed=%d elapsed=%s",
		indexed, failed, time.Since(start).Round(time.Millisecond))
	_ = strings.TrimSpace
}
index_bulk.py — using elasticsearch.helpers.bulk 
from datetime import datetime, timedelta, timezone
from elasticsearch import Elasticsearch
from elasticsearch.helpers import bulk

es = Elasticsearch("http://localhost:9200")

def generate():
    now = datetime.now(timezone.utc)
    for i in range(10_000):
        yield {
            "_op_type": "index",
            "_index":   "books",
            "_id":      f"book-{i}",
            "_source": {
                "title":       f"Generated Title {i}",
                "author":      f"Author {i % 50}",
                "description": f"Description for book {i}",
                "tags":        ["generated", f"batch-{i // 1000}"],
                "price":       9.99 + (i % 40),
                "published":   (now - timedelta(days=i)).date().isoformat(),
                "in_stock":    i % 2 == 0,
                "ingested_at": now.isoformat(),
            },
        }

# chunk_size  -> docs per bulk request
# max_chunk_bytes -> upper bound per request (default 100MB; bring it DOWN)
# request_timeout -> don't let slow batches hang forever
success, errors = bulk(
    es,
    generate(),
    chunk_size=1_000,
    max_chunk_bytes=5 * 1024 * 1024,
    request_timeout=60,
    raise_on_error=False,
)

print(f"indexed={success} errors={len(errors) if isinstance(errors, list) else errors}")
Partial failures are normal. A 200 OK from _bulk does not mean every item succeeded. The response body has a per-item status, and you must inspect it. The helpers above expose OnFailure / errors callbacks for exactly this reason. Log failed _ids and re-queue them — don't just swallow the error.

Step Seven

Searching what you ingested

Ingestion without a search story is just an expensive database. Let's run the two query shapes you'll reach for most often: full-text match on prose, and filtered bool queries that combine exact-match filters with scored musts.

First, with plain curl so you can see the wire format:

shell — a filtered full-text query 
curl -X GET http://localhost:9200/books/_search \
  -H "Content-Type: application/json" \
  -d '{
    "size": 10,
    "query": {
      "bool": {
        "must":   [ { "match": { "description": "programmer craftsmanship" } } ],
        "filter": [
          { "term":  { "in_stock": true } },
          { "terms": { "tags": ["software", "engineering"] } },
          { "range": { "price": { "lte": 50 } } }
        ]
      }
    },
    "sort": [ { "_score": "desc" }, { "published": "desc" } ],
    "_source": ["title", "author", "price", "published"]
  }'

Two things to internalize about that query:

  • must clauses contribute to relevance score. Use them for the user's search terms.
  • filter clauses are yes/no. They don't score, they're aggressively cached, and they're dramatically faster. Anything that doesn't need to affect relevance belongs in filter.

And the same query from each client:

SearchBooks.java 
var response = es.search(s -> s
    .index("books")
    .size(10)
    .query(q -> q.bool(b -> b
        .must(m -> m.match(t -> t.field("description").query("programmer craftsmanship")))
        .filter(f -> f.term(t -> t.field("in_stock").value(true)))
        .filter(f -> f.terms(t -> t.field("tags")
            .terms(v -> v.value(List.of("software", "engineering").stream()
                .map(co.elastic.clients.elasticsearch._types.FieldValue::of).toList()))))
        .filter(f -> f.range(r -> r.number(n -> n.field("price").lte(50.0))))
    ))
    .sort(so -> so.score(sc -> sc.order(co.elastic.clients.elasticsearch._types.SortOrder.Desc)))
    , Book.class);

response.hits().hits().forEach(h ->
    System.out.printf("%.3f  %s — %s%n", h.score(), h.source().title(), h.source().author()));
cmd/search/main.go — using typed API 
package main

import (
	"context"
	"encoding/json"
	"fmt"
	"log"

	"github.com/elastic/go-elasticsearch/v9"
	"github.com/elastic/go-elasticsearch/v9/typedapi/core/search"
	"github.com/elastic/go-elasticsearch/v9/typedapi/types"
)

func main() {
	es, err := elasticsearch.NewTypedClient(elasticsearch.Config{
		Addresses: []string{"http://localhost:9200"},
	})
	if err != nil {
		log.Fatalf("client: %s", err)
	}

	descQuery := "programmer craftsmanship"
	maxPrice  := types.Float64(50.0)
	inStock   := true

	res, err := es.Search().Index("books").Request(&search.Request{
		Size: ptrInt(10),
		Query: &types.Query{
			Bool: &types.BoolQuery{
				Must: []types.Query{
					{Match: map[string]types.MatchQuery{
						"description": {Query: descQuery},
					}},
				},
				Filter: []types.Query{
					{Term: map[string]types.TermQuery{
						"in_stock": {Value: inStock},
					}},
					{Terms: &types.TermsQuery{TermsQuery: map[string]types.TermsQueryField{
						"tags": []string{"software", "engineering"},
					}}},
					{Range: map[string]types.RangeQuery{
						"price": types.NumberRangeQuery{Lte: &maxPrice},
					}},
				},
			},
		},
	}).Do(context.Background())
	if err != nil {
		log.Fatalf("search: %s", err)
	}

	for _, hit := range res.Hits.Hits {
		var src map[string]any
		_ = json.Unmarshal(hit.Source_, &src)
		fmt.Printf("%.3f  %v — %v\n", *hit.Score_, src["title"], src["author"])
	}
}

            ],
        }
    },
    sort=[{"_score": "desc"}, {"published": "desc"}],
    source=["title", "author", "price", "published"],
)

for hit in resp["hits"]["hits"]:
    s = hit["_source"]
    print(f"{hit['_score']:.3f}  {s['title']} — {s['author']}  (${s['price']})")

Step Eight

Best practices that matter in production

If you remember nothing else from this tutorial, remember these. Each one is a mistake that's expensive to fix after you have real data.

01 Always declare an explicit mapping +
Dynamic mapping creates fields at ingest time from what it sees. Every string becomes both text and keyword, every numeric JSON gets guessed (a stray "3" can make your entire field text). That bloats storage, distorts scoring, and breaks aggregations. Set "dynamic": "strict" on anything you own.
02 Know text vs keyword cold +
text is analyzed — tokenized, lowercased, stemmed — and used for full-text search. keyword is the raw string, used for exact match, sorting, aggregations, and term-level filters. Most real fields want both, via a multi-field: text at the top, keyword under .raw.
03 Use index templates and aliases from day one +
Never point your application at a bare index name. Use an alias (e.g. books) that points to a concrete index (books-2026.04). When you need to change the mapping — and one day you will — you reindex into books-2026.05 and atomically switch the alias. Zero downtime, zero code change.
04 Bake metadata into every document +
At a minimum, every document should carry ingested_at (when it hit ES), source_id (who produced it), source_version (schema version of the producer), and updated_at (last mutation in the source system). You'll need these for debugging, replaying ingest, partial reindexing, and proving freshness to stakeholders. Add the fields to the mapping; set them in the client — never rely on ingest pipelines alone.
05 Filter, don't must +
If a clause doesn't contribute to relevance — booleans, IDs, dates, status enums, tenant IDs — put it in the filter array, not must. Filters skip scoring and get cached at the segment level. On a busy cluster this one change can move your p99 latency by an order of magnitude.
06 Right-size your shards +
Target 10–50 GB per primary shard for search-heavy indexes, and up to 30 GB for log/observability indexes. Too many small shards overwhelms the cluster's bookkeeping; too few huge shards hurts query parallelism and rebalancing. Start with number_of_shards: 1 for small indexes and grow via reindex — don't guess 10 because it looks round.
07 Tune refresh_interval for bulk loads +
The default refresh_interval is 1s — every second, ES makes new docs searchable, which costs I/O. During an initial backfill, set it to -1 (disabled) and number_of_replicas: 0, ingest, then turn them back on. You'll often cut load time in half or more.
08 Never send plaintext credentials +
The moment you leave localhost: enable security, use HTTPS, create a dedicated role per application with the minimum privileges it needs (usually write on one index pattern and read on another), and pass credentials via API keys or environment variables — never checked into source. Rotate keys.
