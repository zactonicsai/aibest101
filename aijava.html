Foundation
Overview — Builder Pattern
Fluent object construction · immutability · validation gates

The Builder pattern prevents telescoping constructors by assembling objects field-by-field through a fluent API. The build() call is the single validation gate — the object never exists in an invalid state.

// Fluent builder chain — self-documenting, safe
Document doc = Document.builder()
    .id(UUID.randomUUID().toString())
    .name("Q3-Report.pdf")
    .type("PDF")
    .status("RECEIVED")
    .valid(true)
    .build();  // validates, then constructs
Without Builder
new Doc("id","name",null,"REC",true,null,false) — positional, unreadable, error-prone.
With Builder
Named setters make intent clear. Optional fields remain at default. Validation runs once in build().
  • !
    Always make built objects immutable — use Java records or Lombok @Value.
  • !
    The build() method must be called exactly once per builder instance.
  • !
    Builder ≠ Factory. Builder is step-by-step; Factory creates in one opaque call.
Java 16+
DTOs with Java Records
Immutable data carriers — zero boilerplate, full compiler support

Java Records auto-generate constructor, getters, equals(), hashCode(), toString(). Fields are implicitly private final. Perfect as DTOs, response payloads, and command objects.

public record Document(
    String  id,
    String  name,
    String  type,
    String  status,
    boolean valid,
    Instant createdAt
) {
    // Compact constructor — validation before assignment
    public Document {
        Objects.requireNonNull(name, "name required");
        createdAt = createdAt != null ? createdAt : Instant.now();
    }
    // Derived method
    public boolean isProcessable() { return valid && "RECEIVED".equals(status); }
    public static DocumentBuilder builder() { return new DefaultBuilder(); }
}
  • !
    Records are implicitly final — they cannot be extended.
  • !
    Works natively with Jackson, Spring Data projections, and pattern matching switches.

Jakarta Bean Validation 3.0
Declarative constraint annotations · custom validators · error responses

Annotate record/class fields with constraints. Spring's @Valid on a controller parameter triggers validation automatically — violations produce structured 400 responses.

public record DocumentRequest(
    @NotBlank(message="Name required")
    @Size(max=255)               String name,
    @NotBlank @Size(min=5,max=5000) String content,
    @Pattern(regexp="PDF|DOCX|TXT")  String type,
    @Email                           String submittedBy
) {}
@NotNull
Rejects null only. Empty string passes.
@NotEmpty
Rejects null + empty string/collection.
@NotBlank
Rejects null, empty, AND whitespace. Use for names/content.

Lombok @Builder, @Value, @Singular
Compile-time code generation — zero runtime overhead

Lombok's @Builder generates a complete inner builder class at compile time. @Value makes all fields final with no setters. toBuilder=true enables non-destructive copies.

@Value @Builder(toBuilder = true)
public class DocumentDTO {
    @NonNull         String  id;
    @NonNull         String  name;
    @Builder.Default String  status    = "PENDING";
    @Builder.Default Instant createdAt = Instant.now();
    @Singular        List<String> tags;
}
// Non-destructive copy via toBuilder
DocumentDTO approved = original.toBuilder().status("APPROVED").build();
Async
CompletableFuture — Async in Plain English
Non-blocking pipelines · chaining · error handling · combining results

CompletableFuture lets you run tasks in the background without blocking the current thread. Think of it like placing an order at a coffee shop: you don't stand frozen at the counter — you get a ticket (the Future) and go sit down. When it's ready, you're called back.

⚡
supplyAsync()
Runs a task on a background thread and returns a value.
🔗
thenApply()
Transform the result — like map() for streams.
🧩
thenCompose()
Chain two async operations (flatMap equivalent).
🔀
thenCombine()
Merge two independent futures when both complete.
🛡️
exceptionally()
Recover from errors — provide a fallback value.
🏁
allOf()
Wait for ALL futures to finish before proceeding.
// SIMPLE EXAMPLE — fetch user profile in background
CompletableFuture<UserProfile> profileFuture =
    CompletableFuture
        .supplyAsync(() -> userRepo.findById(userId))   // runs on ForkJoinPool
        .thenApply(user -> mapper.toProfile(user))        // transform result
        .exceptionally(ex -> UserProfile.anonymous());    // fallback on error

// Get the result (blocks here if not ready)
UserProfile profile = profileFuture.join();

// ─────────────────────────────────────────────────
// COMBINE TWO INDEPENDENT TASKS running in parallel
CompletableFuture<String> textFuture   = CompletableFuture.supplyAsync(() -> ocrService.extract(file));
CompletableFuture<String> metaFuture   = CompletableFuture.supplyAsync(() -> metaService.read(file));

CompletableFuture<IndexPayload> combined =
    textFuture.thenCombine(metaFuture,
        (text, meta) -> IndexPayload.builder()
            .text(text).metadata(meta).build());

// ─────────────────────────────────────────────────
// WAIT FOR ALL — process 100 documents concurrently
List<CompletableFuture<Void>> tasks = documents.stream()
    .map(doc -> CompletableFuture.runAsync(() -> process(doc), executor))
    .toList();

CompletableFuture.allOf(tasks.toArray(new CompletableFuture[0])).join();
thenApply vs thenCompose
thenApply(f) — f returns a plain value T.
thenCompose(f) — f returns another CompletableFuture<T>. Use when chaining async calls to avoid nested futures.
get() vs join()
get() throws checked ExecutionException — must handle in try/catch.
join() throws unchecked CompletionException — cleaner in streams and lambdas.
  • !
    Never call get() / join() inside another async stage — that defeats the purpose and can deadlock.
  • !
    Always provide a custom Executor (e.g. ThreadPoolTaskExecutor) in production — the default ForkJoinPool is shared and has no I/O tuning.
  • !
    Use exceptionally() or handle() to prevent silent swallowing of async exceptions.
Spring Async
@EnableAsync · @Async · ThreadPoolTaskExecutor
Spring-managed async methods with custom thread pools

@EnableAsync on your config class tells Spring to proxy any @Async-annotated method and run it on a background thread. Without it, @Async is silently ignored.

// 1. Enable async support + define a custom executor
@Configuration
@EnableAsync
public class AsyncConfig implements AsyncConfigurer {

    @Bean(name = "docExecutor")
    public Executor documentExecutor() {
        ThreadPoolTaskExecutor ex = new ThreadPoolTaskExecutor();
        ex.setCorePoolSize(4);        // min threads always alive
        ex.setMaxPoolSize(16);       // max under burst
        ex.setQueueCapacity(200);   // queue before rejecting
        ex.setThreadNamePrefix("doc-async-");
        ex.setRejectedExecutionHandler(new CallerRunsPolicy());
        ex.initialize();
        return ex;
    }

    // Global uncaught exception handler for @Async methods
    @Override
    public AsyncUncaughtExceptionHandler getAsyncUncaughtExceptionHandler() {
        return (ex, method, params) ->
            log.error("Async error in {}: {}", method.getName(), ex.getMessage());
    }
}

// 2. Annotate service methods — Spring runs them on docExecutor
@Service
public class OcrService {

    @Async("docExecutor")
    public CompletableFuture<String> extractText(Path file) {
        String text = tesseract.doOCR(file.toFile());
        return CompletableFuture.completedFuture(text);
    }
}
⚠ Self-invocation breaks @Async! Calling an @Async method from within the same class bypasses the Spring proxy — the method runs synchronously. Always call @Async methods from a different bean.
✓ Combine @Async with CompletableFuture return type to get both Spring thread management AND a composable async handle back to the caller.
Concurrency
Concurrent Locks & Thread Safety
ReentrantLock · ReadWriteLock · synchronized · AtomicReference · tryLock

When multiple threads access shared data, you need locks to prevent race conditions — two threads reading and writing the same variable simultaneously, producing corrupted results. Java gives you several tools, each with trade-offs.

🔒
synchronized
Simplest. One thread at a time. Built into every Java object. Use for short critical sections.
🗝️
ReentrantLock
Flexible. Supports tryLock(), timed waits, and fairness. Always unlock in finally.
📖
ReadWriteLock
Multiple readers OR one writer. Great for caches — reads are concurrent, writes exclusive.
⚛️
AtomicInteger/Ref
Lock-free for single variables. Uses CPU-level compare-and-swap. Fastest option for counters.
🏎️
StampedLock
Java 8+. Optimistic reads — try without locking, validate afterwards. Highest throughput.
🔄
ConcurrentHashMap
Thread-safe map with stripe-based locking. Far better than synchronizing a HashMap.
// ── ReentrantLock — explicit control with tryLock ──
private final ReentrantLock lock = new ReentrantLock();

public boolean processDocument(String docId) {
    if (!lock.tryLock(500, TimeUnit.MILLISECONDS)) {
        log.warn("Could not acquire lock — skipping {}", docId);
        return false;     // non-blocking fallback
    }
    try {
        // critical section — only one thread here at a time
        doProcess(docId);
        return true;
    } finally {
        lock.unlock();   // ALWAYS unlock in finally
    }
}

// ── ReadWriteLock — concurrent reads, exclusive writes ──
private final ReadWriteLock rwLock = new ReentrantReadWriteLock();
private Map<String, Document> cache = new HashMap<>();

public Document get(String id) {          // many threads can read
    rwLock.readLock().lock();
    try  { return cache.get(id); }
    finally { rwLock.readLock().unlock(); }
}

public void put(String id, Document doc) { // exclusive write
    rwLock.writeLock().lock();
    try  { cache.put(id, doc); }
    finally { rwLock.writeLock().unlock(); }
}

// ── AtomicInteger — lock-free counter ──
private final AtomicInteger processedCount = new AtomicInteger(0);
public void markDone() { processedCount.incrementAndGet(); }   // thread-safe, no lock
  • !
    Always unlock in a finally block. If unlock() is inside try{} and an exception occurs before it, the lock is held forever — deadlock.
  • !
    Never hold a lock while doing I/O (DB calls, HTTP, file reads). Other threads starve waiting. Do I/O outside the lock, then lock only for the write.
  • !
    Lock ordering matters. If Thread A locks X then Y, and Thread B locks Y then X — deadlock. Always acquire locks in the same order everywhere.
Design Pattern
Singleton Pattern & Spring Bean Scope
Classic singleton · enum singleton · Spring singleton scope · double-checked locking

A Singleton ensures only one instance of a class exists in the JVM. In Spring, all beans are singletons by default — one instance per ApplicationContext, shared across all injection points. No need to hand-roll a singleton for Spring beans.

// ── Classic thread-safe singleton (double-checked locking) ──
public class TesseractEngine {
    // volatile prevents CPU caching the partially-constructed object
    private static volatile TesseractEngine INSTANCE;

    private TesseractEngine() { /* expensive init */ }

    public static TesseractEngine getInstance() {
        if (INSTANCE == null) {                  // first check (no lock)
            synchronized (TesseractEngine.class) {
                if (INSTANCE == null)            // second check (with lock)
                    INSTANCE = new TesseractEngine();
            }
        }
        return INSTANCE;
    }
}

// ── Enum singleton — simplest, serialization-safe ──
public enum ElasticClientSingleton {
    INSTANCE;
    private final ElasticsearchClient client = buildClient();
    public ElasticsearchClient get() { return client; }
}

// ── Spring singleton (PREFERRED) — just use @Component ──
@Component  // Spring manages ONE instance, shared via DI
public class DocumentProcessor {
    // Spring injects this everywhere — no getInstance() needed
}
✓ In Spring apps, never hand-roll a singleton. Let Spring manage the lifecycle with @Component / @Bean. Use @Scope("prototype") when you do need a new instance per injection.
  • !
    Singleton beans must be stateless or thread-safe — they're shared across all concurrent requests.
  • !
    Never store request-scoped state (user ID, session data) in a singleton field — use @RequestScope beans or method locals.
Spring MVC
REST Controllers
@RestController · routing · request binding · response patterns · streaming

@RestController = @Controller + @ResponseBody. Every method return value is serialised to JSON automatically. Controllers should be thin — no business logic, just HTTP-to-service translation.

@RestController
@RequestMapping("/api/v1/documents")
@RequiredArgsConstructor
public class DocumentController {

    private final DocumentService   service;
    private final TemporalClient    temporalClient;

    // ── GET all (paged) ──────────────────────────────────
    @GetMapping
    public ResponseEntity<Page<DocumentDTO>> list(
            @RequestParam(defaultValue="0")  int page,
            @RequestParam(defaultValue="20") int size) {
        return ResponseEntity.ok(service.findAll(PageRequest.of(page, size)));
    }

    // ── GET by id ─────────────────────────────────────────
    @GetMapping("/{id}")
    public ResponseEntity<DocumentDTO> getById(@PathVariable String id) {
        return service.findById(id)
            .map(ResponseEntity::ok)
            .orElse(ResponseEntity.notFound().build());
    }

    // ── POST — ingest + trigger Temporal workflow ─────────
    @PostMapping(consumes = MediaType.MULTIPART_FORM_DATA_VALUE)
    public ResponseEntity<WorkflowHandle> ingest(
            @RequestPart("metadata") @Valid DocumentRequest req,
            @RequestPart("file")     MultipartFile file) {

        String workflowId = temporalClient.startOcrWorkflow(req, file);
        URI location = URI.create("/api/v1/documents/status/" + workflowId);
        return ResponseEntity.accepted().location(location).build();
    }

    // ── DELETE ────────────────────────────────────────────
    @DeleteMapping("/{id}")
    @ResponseStatus(HttpStatus.NO_CONTENT)
    public void delete(@PathVariable String id) { service.delete(id); }

    // ── SSE streaming — real-time progress events ─────────
    @GetMapping(value="/stream/{id}", produces=MediaType.TEXT_EVENT_STREAM_VALUE)
    public SseEmitter streamProgress(@PathVariable String id) {
        return service.createProgressStream(id);
    }
}
Return Types
ResponseEntity<T> — full control of status + headers.
T directly — Spring defaults to 200 OK.
void + @ResponseStatus — for 201/204 with no body.
Streaming Options
SseEmitter — Server-Sent Events (one-way push).
StreamingResponseBody — chunked file download.
ResponseBodyEmitter — generic async streaming.
Spring DI
@Component, @Bean, @Service, @Repository
Bean registration · lifecycle · scope · the difference explained simply

Spring's DI container manages objects called beans. You tell Spring what to manage using annotations. Think of it as a box Spring keeps — you ask for things, Spring hands them out.

@Component
Generic stereotype. Put it on any class you own and Spring auto-detects it via component scanning. Use when no more specific stereotype fits.
@Service
Semantically: business logic layer. Technically identical to @Component but communicates intent. Use for service classes.
@Repository
Data access layer. Spring adds exception translation (maps DB exceptions to DataAccessException). Use for DAOs.
@Bean (on @Configuration methods)
For third-party classes you can't annotate — e.g. RestClient, Tesseract, ElasticsearchClient. The @Bean method IS the factory.
// ── @Component / @Service — your own classes ──
@Service                   // same as @Component but clearer
public class DocumentIndexService { ... }

@Repository
public class DocumentRepository { ... }

// ── @Bean — third-party objects Spring can't scan ──
@Configuration
public class InfraConfig {

    @Bean
    public ElasticsearchClient elasticsearchClient(
            @Value("${elastic.url}") String url) {
        RestClient rest = RestClient.builder(
            HttpHost.create(url)).build();
        return new ElasticsearchClient(new RestClientTransport(rest, new JacksonJsonpMapper()));
    }

    @Bean
    public Tesseract tesseract() {
        Tesseract t = new Tesseract();
        t.setDatapath("/usr/share/tessdata");
        t.setLanguage("eng");
        return t;   // Spring holds ONE shared instance
    }

    @Bean
    @Scope("prototype")   // NEW instance every injection point
    public DocumentPipeline documentPipeline() {
        return new DocumentPipeline();
    }
}
  • !
    Component scanning only finds classes in packages at or below your @SpringBootApplication class. Misplace a class outside that package and it's invisible to Spring.
  • !
    Singleton scope is default. One instance for ALL requests. Never store per-request state in a singleton field.
  • !
    Use @Primary or @Qualifier("name") when you have two beans of the same type to tell Spring which to inject.
Spring MVC
@ControllerAdvice — Global Error Handling
Centralised exception handling · structured error responses · @ExceptionHandler · @ResponseStatus

@ControllerAdvice is a single class that intercepts exceptions thrown by any controller in your app. Without it, every controller needs its own try/catch blocks — with it, error handling lives in one place.

Think of it as a safety net stretched under all your controllers. Any unhandled exception falls into it and gets converted to a clean, consistent JSON error response.

@RestControllerAdvice           // @ControllerAdvice + @ResponseBody
public class GlobalExceptionHandler {

    // ── Validation failures (@Valid) → 400 ──────────────
    @ExceptionHandler(MethodArgumentNotValidException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    public ErrorResponse handleValidation(MethodArgumentNotValidException ex) {
        Map<String,String> fields = ex.getBindingResult().getFieldErrors()
            .stream().collect(Collectors.toMap(
                FieldError::getField,
                FieldError::getDefaultMessage));
        return ErrorResponse.of(400, "Validation failed", fields);
    }

    // ── Resource not found → 404 ────────────────────────
    @ExceptionHandler(DocumentNotFoundException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)
    public ErrorResponse handleNotFound(DocumentNotFoundException ex) {
        return ErrorResponse.of(404, ex.getMessage());
    }

    // ── Temporal workflow failure → 502 ─────────────────
    @ExceptionHandler(WorkflowException.class)
    @ResponseStatus(HttpStatus.BAD_GATEWAY)
    public ErrorResponse handleWorkflow(WorkflowException ex) {
        log.error("Workflow error: {}", ex.getMessage());
        return ErrorResponse.of(502, "OCR workflow failed", ex.getCause().getMessage());
    }

    // ── Catch-all → 500 ──────────────────────────────────
    @ExceptionHandler(Exception.class)
    @ResponseStatus(HttpStatus.INTERNAL_SERVER_ERROR)
    public ErrorResponse handleAll(Exception ex, WebRequest request) {
        log.error("Unhandled exception at {}: {}", request.getDescription(false), ex.getMessage(), ex);
        return ErrorResponse.of(500, "Internal server error");
    }
}

// ── Standardised error envelope ──────────────────────
public record ErrorResponse(
    int              status,
    String           message,
    Instant          timestamp,
    Map<String,Object> details
) {
    static ErrorResponse of(int s, String m) {
        return new ErrorResponse(s, m, Instant.now(), Map.of());
    }
    static ErrorResponse of(int s, String m, Object detail) {
        return new ErrorResponse(s, m, Instant.now(), Map.of("detail", detail));
    }
}
✓ Order matters — Spring picks the most specific @ExceptionHandler first. Put catch-all Exception.class last so it doesn't swallow specific cases.
  • !
    Use @RestControllerAdvice (not @ControllerAdvice) when your controllers are REST APIs — it adds @ResponseBody automatically.
  • !
    Log the full stack trace only in the catch-all handler. For known exceptions (404, 400) log at WARN or DEBUG — stack traces are expensive and noisy.
Temporal · OCR · Elastic
Temporal Client → Golang OCR Worker → Elasticsearch
Large-file streaming · Tesseract OCR · full-text indexing · distributed workflows

Temporal is a durable workflow engine. Your Java service is the client that starts workflows. A Go worker polls Temporal for tasks, runs Tesseract OCR on the file, and indexes results into Elasticsearch. If the worker crashes mid-task, Temporal replays from the last checkpoint — no data loss.

// ── Java: Temporal client config (Spring @Bean) ──
@Bean
public WorkflowClient temporalClient(@Value("${temporal.address}") String addr) {
    WorkflowServiceStubs stubs = WorkflowServiceStubs.newServiceStubs(
        WorkflowServiceStubsOptions.newBuilder()
            .setTarget(addr).build());
    return WorkflowClient.newInstance(stubs);
}

// ── Workflow interface (shared contract) ──
@WorkflowInterface
public interface OcrWorkflow {
    @WorkflowMethod
    OcrResult processDocument(OcrRequest req);
}

// ── Service: start workflow + stream progress ──
@Service @RequiredArgsConstructor
public class TemporalDocumentService {

    private final WorkflowClient    client;
    private final StorageService    storage;
    private final Map<String, SseEmitter> emitters = new ConcurrentHashMap<>();

    public String startOcrWorkflow(DocumentRequest req, MultipartFile file) {
        // Stream upload to object storage first
        String storageKey = storage.uploadStreamed(file);

        OcrRequest input = OcrRequest.builder()
            .documentName(req.name())
            .storageKey(storageKey)
            .type(req.type())
            .build();

        OcrWorkflow wf = client.newWorkflowStub(OcrWorkflow.class,
            WorkflowOptions.newBuilder()
                .setTaskQueue("ocr-queue")
                .setWorkflowExecutionTimeout(Duration.ofMinutes(30))
                .build());

        String workflowId = WorkflowClient.start(wf::processDocument, input);

        // Async poll loop — push SSE events to connected browsers
        pollAndEmit(workflowId);
        return workflowId;
    }

    @Async("docExecutor")
    public void pollAndEmit(String wfId) {
        SseEmitter emitter = emitters.get(wfId);
        if (emitter == null) return;
        while (true) {
            WorkflowExecutionInfo info = client.getWorkflowHistory(wfId);
            emitter.send(SseEmitter.event().data(info.getStatus()));
            if (info.isCompleted()) { emitter.complete(); break; }
            Thread.sleep(2000);
        }
    }
}
// ── Go Worker: Temporal activities + Tesseract + Elasticsearch ──

package main

import (
    "context"
    "github.com/otiai10/gosseract/v2"   // Tesseract bindings
    "github.com/elastic/go-elasticsearch/v8"
    "go.temporal.io/sdk/activity"
    "go.temporal.io/sdk/worker"
    "go.temporal.io/sdk/workflow"
)

// ── Activity: OCR extraction ──────────────────────────────────
func ExtractTextActivity(ctx context.Context, req OcrRequest) (string, error) {
    activity.RecordHeartbeat(ctx, "downloading")   // timeout reset
    filePath := downloadFromStorage(req.StorageKey)

    client := gosseract.NewClient()
    defer client.Close()
    client.SetImage(filePath)
    client.SetLanguage("eng")
    text, err := client.Text()
    if err != nil { return "", err }
    activity.RecordHeartbeat(ctx, "ocr-done")
    return text, nil
}

// ── Activity: Elasticsearch indexing ─────────────────────────
func IndexDocumentActivity(ctx context.Context, doc IndexPayload) error {
    es, _ := elasticsearch.NewDefaultClient()
    body, _ := json.Marshal(doc)
    res, err := es.Index("documents",
        bytes.NewReader(body),
        es.Index.WithDocumentID(doc.ID),
        es.Index.WithRefresh("wait_for"),
    )
    defer res.Body.Close()
    return err
}

// ── Workflow: orchestrates activities with retries ────────────
func OcrWorkflow(ctx workflow.Context, req OcrRequest) (OcrResult, error) {
    retryPolicy := &temporal.RetryPolicy{
        MaximumAttempts: 3,
        InitialInterval: time.Second * 5,
    }
    ao := workflow.ActivityOptions{
        StartToCloseTimeout: time.Minute * 10,
        RetryPolicy:         retryPolicy,
    }
    ctx = workflow.WithActivityOptions(ctx, ao)

    var text string
    err := workflow.ExecuteActivity(ctx, ExtractTextActivity, req).Get(ctx, &text)
    if err != nil { return OcrResult{}, err }

    payload := IndexPayload{ID: req.ID, Name: req.Name, Text: text}
    err = workflow.ExecuteActivity(ctx, IndexDocumentActivity, payload).Get(ctx, nil)
    return OcrResult{ID: req.ID, TextLength: len(text)}, err
}

// ── main: register worker ─────────────────────────────────────
func main() {
    c, _ := client.Dial(client.Options{HostPort: "temporal:7233"})
    defer c.Close()
    w := worker.New(c, "ocr-queue", worker.Options{MaxConcurrentActivityExecutionSize: 4})
    w.RegisterWorkflow(OcrWorkflow)
    w.RegisterActivity(ExtractTextActivity)
    w.RegisterActivity(IndexDocumentActivity)
    w.Run(worker.InterruptCh())
}
  • !
    Never pass raw file bytes through Temporal. Workflow inputs are persisted to Temporal's DB — large payloads will OOM and timeout. Stream to object storage first; pass only the key.
  • !
    Call activity.RecordHeartbeat() inside long-running activities (OCR can take minutes). Temporal uses heartbeat to detect stuck workers and trigger retries.
  • !
    Go activities are retried independently — each activity is idempotent. Use the document ID as the Elasticsearch document ID so re-indexing is a no-op.

Builder Pattern — Step Builder & Generic Builder
Advanced compile-time safety patterns
Step Builder — Compile-Time Required Field Enforcement

The Step Builder uses separate interfaces per required field. Skipping a required field is a compile error, not a runtime exception.

public interface IdStep    { NameStep  id(String id); }
public interface NameStep  { TypeStep  name(String name); }
public interface TypeStep  { BuildStep type(String type); }
public interface BuildStep {
    BuildStep status(String s); // optional
    Document  build();
}
// Usage — wrong order = compile error
Document d = DocStepBuilder.newBuilder().id("x").name("f").type("PDF").build();
Generic Abstract Builder (inheritance-safe)

    @SuppressWarnings("unchecked")
    protected B self() { return (B) this; }
    public abstract T build();
}
public class DocBuilder extends AbstractBuilder<Document,DocBuilder> {
    private String name;
    public DocBuilder name(String n) { this.name=n; return self(); }
    public Document build() { return new Document(name); }
}
Key: The self() trick keeps the return type as the concrete subclass, so chained calls return the right type through inheritance.

Concurrent Locks — StampedLock & Deadlock Prevention
StampedLock — Optimistic Reads (Highest Throughput)
private final StampedLock sl = new StampedLock();
private double x, y;

public double distFromOrigin() {
    long stamp = sl.tryOptimisticRead();    // no lock acquired
    double curX = x, curY = y;
    if (!sl.validate(stamp)) {             // was it modified?
        stamp = sl.readLock();              // fall back to real lock
        try   { curX = x; curY = y; }
        finally { sl.unlockRead(stamp); }
    }
    return Math.sqrt(curX*curX + curY*curY);
}
Deadlock Prevention — Lock Ordering
// DEADLOCK RISK — inconsistent order
// Thread A: lock(docA) then lock(docB)
// Thread B: lock(docB) then lock(docA)  ← deadlock!

// FIX — always lock in consistent ID order
void transfer(Document a, Document b) {
    Document first  = a.id().compareTo(b.id()) < 0 ? a : b;
    Document second = first == a ? b : a;
    synchronized(first)  {
        synchronized(second) { doTransfer(a, b); }
    }
}
⚠ Lock + I/O = starvation. Never do database calls, HTTP requests, or file reads inside a lock. Fetch the data first, then lock only for the write to shared state.
