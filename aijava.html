Overview — Why Builder?
Understanding the problem Builder solves in large-scale Java systems

The Builder pattern is a creational design pattern that separates the construction of a complex object from its representation. It is especially valuable when constructors would otherwise require many parameters, some optional, creating ambiguous code that is hard to read and maintain.

In Java 21 enterprise systems — such as IBM's document ingestion pipelines — objects like DocumentRequest, PolicyRecord, or TransactionDTO can have 10–20+ fields. Passing them all via constructor leads to "telescoping constructor" anti-patterns.

Problem Without Builder
new Document("id1", "file.pdf", null, "RECEIVED", true, null, null, false) — unreadable and error-prone.
Solution With Builder
Document.builder().id("id1").name("file.pdf").status("RECEIVED").valid(true).build() — fluent, self-documenting.
Immutability Guarantee
Builder enforces field collection before construction, enabling fully immutable objects once built (records, @Value).
Validation Gate
The build() method is the single point of validation — reject invalid state before the object ever exists.
  • !
    Builder is not the same as the Factory pattern — Builder constructs step-by-step; Factory creates in one call.
  • !
    Always make built objects immutable — use record or @Value (Lombok).
  • !
    Call build() exactly once — reusing a builder after build() can cause shared mutable state bugs.
Core Pattern
Builder Interface
Defining the fluent construction contract with Java interfaces

The DocumentBuilder interface defines the contract for constructing a Document. Each method returns DocumentBuilder itself, enabling method chaining. The final build() call terminates the chain and returns the completed object.

// Builder interface — the construction contract
public interface DocumentBuilder {
    DocumentBuilder id(String id);
    DocumentBuilder name(String name);
    DocumentBuilder type(String type);
    DocumentBuilder status(String status);
    DocumentBuilder valid(boolean valid);
    Document        build();  // terminates the chain
}

// Default implementation (inner or package-private)
class DefaultBuilder implements DocumentBuilder {
    private String id, name, type, status;
    private boolean valid;

    public DocumentBuilder id(String id)     { this.id = id; return this; }
    public DocumentBuilder name(String n)   { this.name = n; return this; }
    public DocumentBuilder type(String t)   { this.type = t; return this; }
    public DocumentBuilder status(String s) { this.status = s; return this; }
    public DocumentBuilder valid(boolean v) { this.valid = v; return this; }

    public Document build() {
        // Validate before constructing
        if (name == null || name.isBlank())
            throw new IllegalStateException("name required");
        return new Document(id, name, type, status, valid);
    }
}
Java 16+
DTO with Java Records
Immutable data carriers using Java's record keyword

Java Records (stable since Java 16) provide compact, immutable data classes with zero boilerplate. They automatically generate constructor, getters, equals(), hashCode(), and toString(). Pairing records with the Builder pattern gives you the best of both: fluent construction + guaranteed immutability.

// Java 21 record — compact immutable DTO
public record Document(
    String  id,
    String  name,
    String  type,
    String  status,
    boolean valid
) {
    // Compact constructor for record-level validation
    public Document {
        Objects.requireNonNull(name, "name must not be null");
        if (name.isBlank()) throw new IllegalArgumentException("name blank");
    }

    // Static factory to enter builder chain
    public static DocumentBuilder builder() {
        return new DefaultBuilder();
    }
}
  • !
    Records are implicitly final — they cannot be subclassed.
  • !
    Record fields are implicitly private and final — set once at construction only.
  • !
    Use compact constructors (no parameter list) to add validation without repeating field assignments.
Jakarta EE
Jakarta Validation (Bean Validation 3.0)
Declarative constraint annotations for input validation

Jakarta Bean Validation 3.0 (formerly Hibernate Validator) provides annotation-driven constraints. When combined with Spring Boot's @Valid, violations automatically return structured 400 Bad Request responses with field-level error messages — no manual if-checks needed.

// Request DTO with validation constraints
public record DocumentRequest(

    @NotBlank(message = "Document name is required")
    @Size(max = 255, message = "Name must not exceed 255 characters")
    String name,

    @NotBlank
    @Size(min = 5, max = 5000, message = "Content must be 5–5000 characters")
    String content,

    @NotNull
    @Pattern(regexp = "PDF|DOCX|TXT", message = "Unsupported type")
    String type,

    @Email
    String submittedBy

) {}

// Spring controller — validation triggered by @Valid
@PostMapping("/documents")
public ResponseEntity<Document> create(
    @Valid @RequestBody DocumentRequest req
) {
    return ResponseEntity.ok(service.create(req));
}
Lombok
Lombok @Builder & @Value
Annotation-driven builder generation — zero boilerplate

Lombok's @Builder annotation generates a complete inner builder class at compile time. Combined with @Value (immutable class) or @Data (mutable), it eliminates hundreds of lines of repetitive code in large domain models. Every IBM Java microservice at scale uses this pattern.

// @Value = immutable; @Builder = auto-generated builder
@Value
@Builder(toBuilder = true)
public class DocumentDTO {
    String  id;
    String  name;
    String  type;
    String  status;
    boolean valid;
    Instant createdAt;

    // Custom default via @Builder.Default
    @Builder.Default
    String status2 = "PENDING";
}

// Usage — Lombok generates all this for free:
DocumentDTO doc = DocumentDTO.builder()
    .id(UUID.randomUUID().toString())
    .name("Q3 Report.pdf")
    .type("PDF")
    .valid(true)
    .createdAt(Instant.now())
    .build();

// toBuilder = true allows non-destructive copies:
DocumentDTO updated = doc.toBuilder().status("PROCESSED").build();

Telescoping Constructor Anti-Pattern

The problem Builder solves is commonly called the telescoping constructor anti-pattern — where you need multiple constructor overloads for every combination of optional parameters.

// BEFORE — telescoping constructors (anti-pattern)
public Document(String id, String name) { ... }
public Document(String id, String name, String type) { ... }
public Document(String id, String name, String type, String status) { ... }
// Grows infinitely for each new optional field

// AFTER — single builder, readable, extensible
Document doc = Document.builder()
    .id("doc-001")
    .name("Report.pdf")
    .build();
Generic Builder Pattern (Reusable)
public abstract class AbstractBuilder<T, B extends AbstractBuilder<T,B>> {
    @SuppressWarnings("unchecked")
    protected B self() { return (B) this; }
    public abstract T build();
}

public class ReportBuilder extends AbstractBuilder<Report, ReportBuilder> {
    private String title;
    public ReportBuilder title(String t) { this.title = t; return self(); }
    public Report build()           { return new Report(title); }
}
Key Reminder: The self() trick in generic builders avoids unchecked casts everywhere and keeps chaining correct through inheritance hierarchies.
Step Builder — Enforcing Required Fields at Compile Time

The Step Builder variant uses separate interfaces per step to enforce that required fields are set before optional ones — a compile-time guarantee rather than a runtime exception.

// Each step interface returns the NEXT step
public interface IdStep   { NameStep id(String id); }
public interface NameStep { TypeStep name(String name); }
public interface TypeStep { BuildStep type(String type); }
public interface BuildStep {
    BuildStep status(String s);  // optional
    Document  build();
}

// Compiler forces correct order — missing id() won't compile
Document d = DocumentStepBuilder
    .newBuilder()   // returns IdStep
    .id("doc-1")    // returns NameStep
    .name("file")  // returns TypeStep
    .type("PDF")   // returns BuildStep
    .build();
✓ Use Step Builder when certain fields are truly required and you want compile-time enforcement rather than relying on runtime validation in build().
Common Pitfalls & How to Avoid Them
⚠ Pitfall 1 — Shared Builder Instance: Never store a builder as a field and reuse it across threads. Builders are NOT thread-safe by default.
⚠ Pitfall 2 — Silent Null Fields: If you don't validate in build(), null fields silently produce broken objects. Always assert required fields.
⚠ Pitfall 3 — Mutable Collections: If a field is a List, copy it defensively: List.copyOf(tags) in build().
⚠ Pitfall 4 — @Builder with @NonNull (Lombok): Lombok's @NonNull generates null checks in the setter, but only if you also use @Builder — make sure both annotations are present.
// Defensive copy for collection fields
public Document build() {
    Objects.requireNonNull(name, "name");
    Objects.requireNonNull(type, "type");
    List<String> safeTags = tags == null
        ? List.of()
        : List.copyOf(tags);  // immutable copy
    return new Document(id, name, type, status, valid, safeTags);
}
Builder Interface — Advanced Examples
Builder with Conditional Logic

Builders can encapsulate complex conditional logic before constructing the final object, keeping that complexity out of callers.

public Document build() {
    // Auto-generate ID if not provided
    if (id == null)
        id = UUID.randomUUID().toString();

    // Infer type from name extension if not set
    if (type == null && name != null) {
        type = name.endsWith(".pdf")  ? "PDF"  :
               name.endsWith(".docx") ? "DOCX" : "UNKNOWN";
    }

    // Validation gate
    Objects.requireNonNull(name, "name is required");
    if ("UNKNOWN".equals(type))
        throw new IllegalStateException("Unsupported document type");

    return new Document(id, name, type,
        status != null ? status : "RECEIVED", valid);
}
Builder Pattern in Spring Services
@Service
public class DocumentService {

    public Document ingest(DocumentRequest req) {
        return Document.builder()
            .id(UUID.randomUUID().toString())
            .name(req.name())
            .type(req.type())
            .status("RECEIVED")
            .valid(isValid(req))
            .build();
    }

    private boolean isValid(DocumentRequest r) {
        return r.name() != null
            && !r.name().isBlank()
            && r.content().length() >= 5;
    }
}
Key Reminder: Keep business logic (like isValid) out of the Builder itself. The Builder handles construction; the Service handles business rules.
Java Records — Advanced Patterns
Records vs. Traditional POJOs
Traditional POJO (150+ lines)
Requires manual: constructor, getters, equals(), hashCode(), toString(), copy constructors, builder inner class.
Java 21 Record (5 lines)
All of the above generated automatically. Immutable by default. Works perfectly with Jackson, JPA projections, and Spring.
Record with Custom Accessor & Derived Field
public record Document(
    String   id,
    String   name,
    String   type,
    String   status,
    boolean  valid,
    Instant  createdAt
) {
    // Compact constructor — runs before auto-assignment
    public Document {
        Objects.requireNonNull(id, "id");
        name = name == null ? "Untitled" : name.strip();
        createdAt = createdAt != null ? createdAt : Instant.now();
    }

    // Custom accessor (overrides generated one)
    public String name() { return name.toUpperCase(); }

    // Derived / computed method
    public boolean isProcessable() {
        return valid && "RECEIVED".equals(status);
    }

    // Static factory
    public static DocumentBuilder builder() {
        return new DefaultBuilder();
    }
}
Key Reminder: Records work natively with pattern matching in Java 21 switch expressions and instanceof checks — ideal for routing logic based on document type.
⚠ Records cannot extend classes (they implicitly extend java.lang.Record). Use interfaces for polymorphism with records.
Jakarta Validation — Advanced Constraints
Custom Constraint Annotation

When built-in annotations aren't enough, define your own custom constraint — for example, validating that a document name matches an allowed naming convention.

// 1. Define the annotation
@Documented
@Constraint(validatedBy = ValidDocNameValidator.class)
@Target({ElementType.FIELD})
@Retention(RetentionPolicy.RUNTIME)
public @interface ValidDocName {
    String   message()  default "Invalid doc name format";
    Class[]  groups()   default {};
    Class[]  payload()  default {};
}

// 2. Implement the validator
public class ValidDocNameValidator
    implements ConstraintValidator<ValidDocName, String> {

    private static final Pattern PATTERN =
        Pattern.compile("^[A-Za-z0-9_\\-]+\\.(pdf|docx|txt)$");

    public boolean isValid(String v, ConstraintValidatorContext ctx) {
        return v != null && PATTERN.matcher(v).matches();
    }
}

// 3. Use it like any other annotation
public record DocumentRequest(
    @ValidDocName String name,
    @NotBlank     String content
) {}
Global Exception Handler for Validation Errors
@RestControllerAdvice
public class ValidationExceptionHandler {

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<Map<String,String>> handle(
        MethodArgumentNotValidException ex) {

        Map<String,String> errors = ex.getBindingResult()
            .getFieldErrors().stream()
            .collect(Collectors.toMap(
                FieldError::getField,
                FieldError::getDefaultMessage
            ));

        return ResponseEntity.badRequest().body(errors);
    }
}
Key Reminder: @NotNull rejects null only. @NotEmpty rejects null and empty string/collection. @NotBlank rejects null, empty, AND whitespace-only strings. Choose carefully.
Lombok — Advanced Patterns & Pitfalls
@Builder with Validation via @Builder.ObtainVia
@Value
@Builder
public class DocumentDTO {

    @NonNull                    // Lombok null check in setter
    String id;

    @NonNull
    String name;

    @Builder.Default            // Default if not set by caller
    String status = "PENDING";

    @Builder.Default
    Instant createdAt = Instant.now();

    @Singular                   // Generates addTag() / clearTags()
    List<String> tags;
}

// Usage with @Singular — add items one by one
DocumentDTO doc = DocumentDTO.builder()
    .id("d-001")
    .name("Report.pdf")
    .addTag("finance")
    .addTag("q3")
    .build();
Non-Destructive Copies with toBuilder
// toBuilder = true lets you clone and modify without mutation
DocumentDTO original = DocumentDTO.builder()
    .id("d-001").name("Draft.pdf").build();

// Produce a new object with status changed — original untouched
DocumentDTO approved = original.toBuilder()
    .status("APPROVED")
    .build();

// original.status() still == "PENDING" — fully immutable
⚠ Key Lombok Pitfalls
⚠ @Data vs @Value: @Data generates mutable getters/setters. @Value makes all fields final and generates no setters. Prefer @Value for DTOs.
⚠ @Builder.Default doesn't work with @NonNull: Lombok generates separate code paths; mixing them can cause NullPointerException in edge cases. Test explicitly.
⚠ IDE annotation processing: Lombok requires the annotation processor enabled in your IDE. Missing this causes "cannot find symbol" errors that aren't real compilation errors.
Key Reminder: Lombok is a compile-time tool — it adds zero runtime overhead. All generated code is visible in the compiled .class files and in your IDE's "delombok" view.
